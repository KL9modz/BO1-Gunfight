The mod replaced the native round timer with a CUSTOM clock (`gf_startRoundClock` in `_gf_rounds.gsc`): it holds `level.timeLimitOverride = true` for the whole round and drives the HUD via `setGameEndTime`. Reason it's custom: the engine's `timeLimitClock` fires a hardcoded "time running out" sequence (announcer VO + TIME_OUT music + beeps) at ~30s remaining, which misfires on the 45s rounds and has no dvar to retune.

The prematch is the engine's native per-round prematch (re-armed each round by setting `level.prematchPeriod` in `onStartGameType`; durations `scr_gf_match_prematch_seconds` 20 / `scr_gf_prematch_seconds` 7). `gf_tryActivateRound` fires ~0.2s INTO the prematch (players spawn frozen during the countdown, and that first spawn triggers it), so `gf_startRoundClock` was starting DURING the countdown → `timeLimitOverride=1` defeated the engine's `checkTimeLimit` prematch hide (it returns at line 1 when override is set) AND burned ~7s of round time. Fix: `gf_tryActivateRound` does `level waittill("prematch_over")` before `gf_startRoundClock`. (The old custom `gf_roundStartCountdown`/freeze path and its `scr_gf_native_prematch` toggle were removed once native was proven — native is now the only path.)

**Why:** you CANNOT cleanly control the round-timer HUD during `inPrematchPeriod` from script. Any visible `setGameEndTime` value flickers (the engine drives the timer in that window and fights re-pushes), and even a `setGameEndTime(0)` hide-loop flickers. Dead ends tried this session: one-time hide, freeze-via-re-push (moving target = flicker, also broke OT), 20Hz hide-loop (still flickered). The only clean behavior is SD's: don't touch `timeLimitOverride` during prematch and let the engine hide the timer itself.

**How to apply:** to control the round timer around the prematch, gate the custom clock to start at `prematch_over`; never push `setGameEndTime` while `inPrematchPeriod`. SD timer model (the reference): native prematch is MATCH-START only — the engine zeroes `level.prematchPeriod` every round and only refills it in the once-per-match `!game["gamestarted"]` block, so re-arm it in `onStartGameType` for a per-round countdown. SD bomb-plant pause = `pauseTimer()` + `timeLimitOverride=true` + `setGameEndTime(now + bombTimer)` (swaps the round clock for the bomb countdown; never resumes). GF's OT pause borrows this pattern but hides (`setGameEndTime(0)`) instead of swapping, and resumes on capture-break. Related: [[onprecache-once-per-match-loadfx-wiped]].

**UPDATE 2026-07-01 (commit 7c9f02c) — roster-gated activation inserted between `prematch_over` and clock start.** `gf_tryActivateRound` now: waittill(prematch_over) → `pauseTimer()` + dud-disable IMMEDIATELY (the unpaused stock `timeLimitClock` starts inside its hardcoded [40,60]s `match_ending_soon` band on a 45s round → premature last-round winning/losing VO; and pausing freezes `getTimePassed()` → grenade duds — both must be handled BEFORE any hold) → bounded 8s poll until every teamed player `hasSpawned` (`setGameEndTime(0)` hides the clock during a real hold; safe here because prematch is over) → `gf_closeGraceEarly` (grace closes when roster is in, 3s floor for team-select idlers who are invisible to the poll; `level.gracePeriod` back to stock 15 as ceiling; close mirrors stock: flag + threaded `updateTeamStatus` or a during-hold wipe is never noticed) → `gf_updateAutoTeamMode` → `gf_startRoundClock`. Engine facts learned: stock `waitForPlayers()` is an EMPTY stub; `maySpawn` locks any unspawned player out for the whole round once grace closes (`everExisted` both teams + !hasSpawned); the vendored `bot_wait_for_host` deterministically burned 10s on dedis (no host ever exists) — now skipped when `getDvarInt("dedicated") >= 1`.
