**RECOVERY (if MOUSE2/ADS is dead in-game): open the client console and run `exec autoexec`.** That re-applies `unbind K` + the MOUSE2 ADS bind from `%localappdata%\Plutonium\storage\t5\players\autoexec.cfg` to the live bind table. Plutonium does NOT auto-exec it, so it's a manual per-session step. The game strips custom MOUSE2 binds from the GLOBAL `config_mp.cfg` on every launch (the per-mod `mods/mp_gunfight/config_mp.cfg` copy survives better); editing the cfg files while the game is RUNNING is futile — it overwrites them from its in-memory bind table on save. So: either edit cfgs with the game fully closed, or set the bind live in console. (2026-06-16: user hit this again after rebinding ADS to K in the controls menu, which cleared MOUSE2; plain `bind MOUSE2 "+speed_throw"` is the durable form since the game preserves non-compound binds across launches.)

BO1/T5 blocks ADS while the sprint command (`+breath_sprint`, SHIFT) is held — you must physically release sprint before aiming. Client-side fix = compound binds that clear the opposing input on press.

**Final kept bind (2026-06-15) — MOUSE2 only:**
```
bind MOUSE2 "+speed_throw; -breath_sprint; -sprint"            // ADS clears sprint -> can aim w/o releasing Shift
bind SHIFT  "+breath_sprint"                                   // left at DEFAULT
```
The "sprint wins" SHIFT compound bind (`+breath_sprint; -speed_throw; -toggleads_throw`) worked but was DROPPED by user choice: combined with the config_mp.cfg churn (game rewrites it per mod-vs-vanilla context and silently drops custom binds whose in-memory value isn't present at save time), keeping two custom compound binds persistent was too fragile. User accepts the residual quirk (tapping sprint mid-ADS un-ADSs you). To persist a bind past the game's rewrite: set it in console so the in-memory value is correct at the next save.

**Three non-obvious gotchas (cost many iterations):**
1. The bind MUST start with the `+command`. The engine only auto-emits the release (`-speed_throw`) on key-up when the bind string *begins* with that `+command`. Starting with `-breath_sprint` → no release emitted → stuck in ADS.
2. Only HOLD ads (`+speed_throw`) works, NOT toggle-ADS (`+toggleads_throw`). The toggle command needs its keynum parm to register the press; the compound bind starves it → "can't ADS". So these binds force Hold-ADS regardless of the menu toggle setting.
3. **The trailing absorber token is REQUIRED.** The engine appends the pressing key's `<keynum> <time>` to the END of the whole bind string, so it lands on the LAST token. The button-release (`IN_KeyUp`) is a NO-OP when that keynum doesn't match the key currently holding the button (e.g. `-breath_sprint` gets MOUSE2's keynum but SHIFT holds sprint → mismatch → sprint never clears → "back to where I started"). A `-command` with NO args force-releases. So put an inert trailing `-command` (`-sprint` / `-toggleads_throw`) to soak up the appended keynum, letting the real release token run clean. The 2-token form `"+speed_throw; -breath_sprint"` LOOKS right and even saves to config, but silently fails this way.

Trade-off of both binds: after the cross-cancel, the other action won't re-engage until you release+repress its key (engine doesn't auto-repeat held `+commands`). Acceptable. SHIFT bind also loses hold-breath-while-scoped (fine for gunfight).

**Config location — Plutonium client does NOT read the Steam-install `players/` folder.** It reads/writes:
- `%localappdata%\Plutonium\storage\t5\players\config_mp.cfg` (global, loaded at startup)
- `%localappdata%\Plutonium\storage\t5\players\mods\<mod>\config_mp.cfg` (per-mod)
These are "// generated by Call of Duty, do not modify" and the running game rewrites them from scratch. **CONFIRMED: the global config_mp.cfg silently DROPS the MOUSE2 compound bind on restart** — user had to re-type `bind MOUSE2 ...` in console every launch. The per-mod copy kept it but isn't what loads at startup, so ADS was dead on every fresh restart.

**Persistence fix (2026-06-15, FINAL):** the bind lives in `%localappdata%\Plutonium\storage\t5\players\autoexec.cfg` (a hand-made file the game never rewrites; the Steam-install `Call of Duty Black Ops\players\autoexec.cfg` is NOT read by the client). **CONFIRMED: Plutonium T5 does NOT auto-exec autoexec.cfg** — user had to `exec autoexec` manually each session.
